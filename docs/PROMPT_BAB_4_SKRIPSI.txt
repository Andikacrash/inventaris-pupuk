--------------------------------------------------------------------------------
A. IDENTITAS PROYEK (STACK & ARSITEKTUR)
--------------------------------------------------------------------------------
- Nama domain aplikasi: inventaris dan penjualan barang (fokus pupuk; kategori
  juga mendukung alat pertanian, herbisida, pestisida setelah konsolidasi data).
- Backend: Laravel 12, PHP ^8.2.
- Otentikasi: sesi web (login Blade) + API Laravel Sanctum (`personal_access_tokens`);
  middleware alias `role` memeriksa peran pengguna.
- Peran (enum pada users.role): admin, kasir, manager (default migrasi: kasir).
- Frontend: Vite, React 19, Axios; styling Tailwind CSS 4 + Bootstrap 5;
  visualisasi: Chart.js, react-chartjs-2.
- Laporan unduhan: barryvdh/laravel-dompdf (PDF), maatwebsite/excel (Excel).
- Basis data: MySQL/MariaDB (umum dengan Laravel); skema dari migration Laravel.

Arsitektur umum (jelaskan dalam teks): klien (browser, SPA React pada halaman
tertentu) ↔ HTTP (web routes + API routes) ↔ Controller / Service ↔ Eloquent
Models ↔ DBMS.

--------------------------------------------------------------------------------
B. AKSES FITUR MENURUT PERAN (RINGKASAN routes/web.php)
--------------------------------------------------------------------------------
- Login / logout: publik login; logout auth.
- Registrasi pengguna: hanya admin (`/register`).
- Redirect `/` setelah login: ke `/sales` (kasir).
- Dashboard ringkasan `/dashboard`: admin, kasir, manager.
- Produk CRUD `/products*`: hanya admin.
- Supplier `/suppliers`: admin (GET index, POST store).
- Penjualan POS `/sales`: admin, kasir.
- Hutang `/debts*`, cicilan terkait: admin, kasir.
- Laporan `/reports`, `/stock-reports`: admin, manager.

--------------------------------------------------------------------------------
C. API & MODUL (RINGKASAN routes/api.php — tanpa mengarang path lain)
--------------------------------------------------------------------------------
Prefix umum API: `/api/...` (sesuai konfigurasi Laravel).

Dashboard (middleware web + auth):
  GET dashboard/summary, sales-chart, sales-comparison, sales-by-period,
  sales-analysis, month-comparison, debt-management, top-products,
  category-chart, low-stock-alerts, recent-sales.

POS (middleware web + auth, prefix pos):
  GET pos/search-products, pos/product-by-barcode
  POST pos/sales
  GET pos/sale-history, pos/sale-detail/{id}
  DELETE pos/cancel-sale/{id}

Laporan (prefix reports — unduhan & preview; edit penjualan untuk UI laporan):
  GET reports/download/sales/pdf|excel, stock/pdf|excel, product/excel
  GET reports/preview/sales, preview/stock
  GET reports/sales-report, stock, stock-movement, supplier, category,
      top-selling-products, sales, sales-analytics, sales/{id}
  PUT reports/sales/{id}, DELETE reports/sales/{id}

Stok (prefix stock):
  GET stock/history, low-stock-alerts, prediction
  POST stock/adjust, stock/bulk-adjust

Produk REST: apiResource products (ProductController API).

Kategori untuk filter: GET /categories (JSON dengan sort_order, withCount products).

Sanctum: GET /user dengan auth:sanctum.

4.3 Perancangan

4.3.1 Perancangan Proses Sistem
  (1) Use case diagram — deskripsi naratif + daftar aktor dan use case per
      modul (autentikasi, master produk & supplier, POS, hutang & cicilan,
      stok, laporan & unduhan, dashboard). Jelaskan relasi include/extend
      bila dipakai.
  (2) Activity diagram — minimal TIGA alur terpisah dalam teks langkah
      terstruktur (mulai, aktivitas, decision, akhir):
      - Penjualan/POS termasuk validasi stok dan pembatalan transaksi.
      - Pencatatan mutasi stok / penyesuaian stok.
      - Pembuatan atau pratinjau / unduhan laporan (PDF/Excel) dengan filter periode.

4.3.2 Perancangan Struktur Data
  (1) Perancangan konseptual — entitas & atribut domain (bahasa bisnis).
  (2) Entity Relationship Diagram — uraian teks entitas dan kardinalitas.
  (3) Relasi antar tabel — tabel ringkas: Tabel | Relasi | Tabel terkait | Keterangan.
  (4) Struktur tabel — untuk setiap tabel bisnis utama pada bagian D di atas:
      nama kolom, tipe logis, PK/FK, keterangan singkat. Sertakan catatan
      kolom Sale yang ada di model namun tidak tercantum di migrasi 153039.

4.3.3 Perancangan Antarmuka Pengguna
  — Inventaris layar sesuai rute: login, dashboard, produk, supplier, penjualan,
    hutang, laporan, laporan stok. Untuk masing-masing: tujuan layar, peran
    yang berhak, komponen UI utama (form, tabel, grafik, unduhan).

4.4 Implementasi

4.4.1 Implementasi Basis Data pada DBMS
  — DBMS target, pembuatan skema lewat `php artisan migrate`, konsistensi
    dengan 4.3.2, indeks/unik (invoice_number, barcode, email). Prosedur
    verifikasi skema (opsional): migrasi status, dump skema.

4.4.2 Implementasi Kode Program
  — Lapisan: routes, middleware peran, controller (web + Api: Pos, Report,
    Stock, Dashboard, Product, Debt, Installment, Auth), model Eloquent,
    view Blade + aset Vite/React. Jelaskan SATU alur request–response untuk
    POS dan SATU untuk unduhan laporan (tanpa tempel kode ratusan baris;
    boleh cuplikan pseudokode atau nama kelas/metode).

4.5 Pengujian

4.5.1 Pengujian Metode HCD (Human-Centered Design)
  — Tahapan HCD dan hubungannya dengan kebutuhan pengguna, perancangan UI,
    dan iterasi (empati, definisi, ide, prototipe, uji).

4.5.2 Pengujian Fungsionalitas (Black Box Testing)
  — Tabel minimal 18 kasus uji dengan kolom:
    ID, Modul, Deskripsi, Langkah uji, Data uji, Hasil diharapkan, Hasil aktual (kosong).
    Cakupan wajib: login gagal/benar, akses ditolak per peran, CRUD produk,
    supplier, transaksi POS (termasuk kredit jika relevan), hutang &
    pembayaran, cicilan, pembatalan penjualan, laporan unduh PDF/Excel,
    endpoint dashboard ringkasan, penyesuaian stok.

4.5.3 Pengujian System Usability Scale (SUS)
  — Jelaskan 10 pertanyaan SUS (versi Bahasa Indonesia netral), skala 1–5,
    rumus skor SUS 0–100, interpretasi rentang skor menurut literatur umum.
  — Contoh format lembar respon untuk responden.
